# Контракт Faker LLM ↔ нагрузочный скрипт (in-band, односторонний)

> Реализованный контракт со стороны Faker. Эталон грамматики маркера — функции
> `DirectiveMarker`/`ParseMarker` нагрузочного тестера (LoadTesting `internal/faker/marker.go`);
> грамматика Faker (`PromptDirectivePolicy`) повторяет их 1-в-1, чтобы round-trip совпадал.

## 1. Назначение

В режиме `faker` каждый запрос несёт **директиву** с ожидаемым типом ответа. Faker
синтезирует ответ этого типа (независимо от пула), а скрипт детерминированно проверяет
соответствие под нагрузкой. Это отделяет проверку *корректности обвязки* (шлюз, лицензия)
от качества живой модели.

## 2. Транспорт: один in-band канал внутри текста сообщения

Цепочка запроса: `клиент → Enterprise License Server (ЛК) → ohmycode → Gateway → Faker`.

> **Почему именно текст сообщения.** ЛК пересобирает запрос/ответ из типизированных DTO и
> **срезает всё нестандартное в обе стороны**: тело запроса (`x_faker`, `request_id`,
> `max_tokens`), кастомные заголовки и любые расширения тела ответа. Единственный канал,
> переживающий ЛК, — **содержимое сообщения** (`messages[].content` на вход,
> `content`/`reasoning_content`/`tool_calls` на выход). Поэтому директива едет **внутри текста**
> запроса как маркер, а ответ остаётся **чистым** OpenAI.

**Канал односторонний: эха в ответе НЕТ.** Время апстрима задаёт сам клиент (в директиве),
Faker применяет его детерминированно — поэтому оверхед шлюза клиент считает из *запрошенного*
тайминга, без обратного эха (§8). В ответе нет ни `request_id`, ни `x_faker.applied_timing`.

### Маркер директивы

В текст user-сообщения дописывается один самодостаточный токен:

```
[[faker:type=<type>;status=<int>;ttft=<ms>;itl=<ms>;total=<ms>]]
```

Пример тела запроса (обычный chat/completions, маркер — в тексте):

```jsonc
{
  "model": "...",
  "stream": true,
  "stream_options": { "include_usage": true },
  "messages": [
    { "role": "user", "content": "lorem ipsum ... [[faker:type=normal;ttft=300;itl=20;total=2000]]" }
  ]
}
```

**Правила разбора** (реализованы в `PromptDirectivePolicy`, эталон — `faker.ParseMarker`):
- берётся **первое** вхождение `[[faker:` … `]]`; последующие маркеры игнорируются;
- пары разделены `;`, ключ и значение — первым `=`;
- ключи регистронезависимы и обрезаются по пробелам; неизвестные ключи игнорируются;
- нечисловое значение → `0`;
- если `type` отсутствует/пуст — это **не директива** (обычный пул-ответ).

| ключ | тип | для каких типов | смысл |
|---|---|---|---|
| `type` | enum | все | какой ответ произвести (список закрыт, см. §5) |
| `status` | int | `error` | HTTP-статус ошибки (если опущен — **400**; код/текст — на усмотрение Faker) |
| `ttft` | int (мс) | `normal`/`thinking`/`tool_call`/`empty` | задержка до первого токена |
| `itl` | int (мс) | `normal`/`thinking` | пауза между токенами |
| `total` | int (мс) | `normal`/`thinking` | целевая длительность стрима (§4) |

> **Синтез независим от пула.** Тип/статус/тайминг полностью заданы маркером: `error` →
> синтетическая HTTP-ошибка, остальные → синтезированный ответ (`SyntheticEntryBuilder`). Привязка
> к записям пула не используется (через ЛК она давала 500 → ретраи → таймаут).

**Флаг `stream` тела запроса Faker чтит:** `true` → SSE; `false` → один JSON.

`thinking.min_tokens` и `tool_call.name` через ЛК ненаблюдаемы и в маркере **не передаются** —
Faker берёт свои дефолты.

## 3. Семантика директивы (ADT)

Директива — **размеченное объединение**: `type` выбирает вариант, значимы только его поля.

```
normal     ttft, itl, total                 — стрим content-токенов, длительность по §4
thinking   ttft, itl, total                 — reasoning-блок, затем content
tool_call  ttft                             — один вызов функции
empty      ttft                             — структурно валидный ответ без content
error      status                           — HTTP-ошибка с этим статусом (дефолт 400)
timeout    (без тайминга)                   — ответ не завершается до клиентского таймаута
```

Если тайминг не задан — Faker применяет свои дефолты.

## 4. Размер ответа: задаётся длительностью

Размер успешного стримового ответа (`normal`/`thinking`) задаётся **длительностью**, а не
числом токенов. Темп — `ttft` (до первого токена) и `itl` (между токенами); число content-токенов
`n` Faker выводит так, чтобы стрим длился ~`total`:

```
n = round((total − ttft) / itl) + 1        (при itl > 0; иначе n = 1)
```

Пример: `total=2200, ttft=200, itl=20` → `n = round((2200−200)/20)+1 = 101` токен, стрим ~2.2с.

**Детерминизм обязателен.** Поскольку эха нет, клиент считает «время апстрима» равным
запрошенному `total` (для `tool_call`/`empty` — `ttft`). Это работает только если Faker
действительно выдерживает `ttft`/`itl` — иначе его собственная задержка уедет в «оверхед шлюза».
Faker применяет тайминг **точно** (фиксированные значения, без min/max-рандома: по одному токену
на чанк → одна `itl`-пауза на токен); точность сверяется отдельным прямым прогоном в Faker
(где `E2E ≈ время Faker`).

**`max_tokens` в faker не используется** (его всё равно срезает ЛК): размер определяет директива.
В `real`-режиме директивы нет, и `max_tokens` — единственный регулятор длины.

`thinking` так же управляется длительностью: токены делятся на reasoning-блок и content.
`tool_call` — один токен-вызов. `empty` — ноль content-токенов (значим только `ttft`).

## 5. Поведение и проверки по типам (6 типов)

| type | что Faker производит | что клиент считает корректным |
|---|---|---|
| `normal` | успешный стрим content-токенов (длительность по §4) | не ошибка И ответ непустой |
| `thinking` | reasoning-блок (`delta.reasoning_content`), **затем** content | не ошибка И reasoning-блок наблюдался |
| `tool_call` | один вызов функции, `finish_reason: "tool_calls"` | не ошибка И есть вызов |
| `error` | HTTP-ответ со статусом `status` | ответ-ошибка **нужного класса** (точный код не сверяется) |
| `timeout` | не завершать ответ дольше клиентского таймаута | сработал клиентский таймаут |
| `empty` | структурно валидный ответ без content-токенов | не ошибка |

> **Статус ошибки сверяется по классу, а не точно.** ЛК **перемаппит** статус апстрима в свою
> таксономию (напр. `418→502`) и переписывает тело ошибки своим конвертом. Поэтому клиент
> проверяет лишь, что пришла ошибка (`status ≥ 400`), а не конкретный код. Тестирование
> rate-limit — это `error;status=429`; «медленный» ответ — `normal` с большим `total`.

Ожидаемый «неуспех» для `error`/`timeout` **не** считается перфоманс-ошибкой в метриках.
Список типов закрыт: неизвестный `type` маркером не считается директивой (пул-ответ).

## 6. Формат ответа (чистый OpenAI, без эха)

**Стриминг (`stream: true`):**
- `Content-Type: text/event-stream`; каждый чанк — `data: {json}\n\n`; поток завершается строкой
  `data: [DONE]`.
- content → `choices[0].delta.content`; reasoning → `choices[0].delta.reasoning_content`;
  вызов функции → `choices[0].delta.tool_calls`.
- **`finish_reason` обязателен** в финальном чанке, если выдан хотя бы один токен (`"stop"`; для
  `tool_call` — `"tool_calls"`).
- Финальный usage-чанк несёт `usage.completion_tokens` = число фактически отданных токенов
  (`stream_options.include_usage` клиент включает всегда).
- **Никаких `request_id` / `x_faker` в чанках.**

**Нестриминг (`stream: false`):** один JSON `application/json` с `choices[0].message`.

**Ошибки (`error`):** HTTP-статус = запрошенный, тело — обычный
`{"error":{"message","type","code"}}` без эха (ЛК его перепишет; клиент читает только класс).

## 7. Эхо и request-id — НЕ используются

Контракт односторонний. Клиент **не шлёт** `request_id` и **не ожидает** никакого эха в теле
ответа: всё это срезается ЛК. Корреляция «ответ↔запрос» обеспечивается транспортом (один ответ на
одно соединение) и косвенно — тип-конформностью (§5).

## 8. Оверхед шлюза — из запрошенного тайминга

Поскольку фактическое время Faker через ЛК ненаблюдаемо, клиент считает:

```
overhead = E2E − запрошенный_upstream
           где запрошенный_upstream = total (normal/thinking) либо ttft (tool_call/empty)
```

- Считается только для **успешных 2xx-ответов** с заданным таймингом. Для `error`/`timeout`/без
  тайминга оверхед не вычисляется.
- Отрицательный оверхед клампится в 0 (артефакт округления мс против мкс-точного E2E).
- В `real`-режиме директивы нет → оверхед не вычисляется.
- Корректность допущения «запрошенное ≈ фактическое» проверяется **калибровкой**: прямой прогон в
  Faker, где `E2E ≈ время Faker` (оверхеда нет).

## 9. Чего нельзя проверить через ЛК (пределы тракта, не дизайна)

- **точный HTTP-статус ошибки** — ЛК ремапит (проверяем только класс);
- **`error`/`timeout`/`empty`**: оверхед недоступен (нет content-ответа с известным таймингом);
- **идентичность Faker** (`id`/`model` ответа ЛК подменяет своими);
- **cross-talk** на error/timeout/empty (нет корреляционного маркера).

При **прямом** прогоне в Faker (минуя ЛК) ограничений нет — это путь калибровки.

## 10. Инвариант конфигурации для `timeout`

Faker удерживает ответ дольше клиентского `request_timeout`. Клиент считает **сработавший
таймаут успехом** для `timeout`, а вовремя пришедший ответ — провалом. ⚠️ Поэтому
`request_timeout` в конфиге обязан быть **меньше** времени удержания.

## 11. Примеры (wire `openai`)

### A. `normal`, стриминг, длительность ~2.2с

```http
POST /v1/chat/completions HTTP/1.1
Authorization: Bearer <api_key>
Content-Type: application/json
Accept: text/event-stream

{ "model": "stub-model",
  "stream": true, "stream_options": { "include_usage": true },
  "messages": [{ "role": "user",
    "content": "lorem ipsum ... [[faker:type=normal;ttft=200;itl=20;total=2200]]" }] }
```

```
HTTP/1.1 200 OK
Content-Type: text/event-stream

data: {"object":"chat.completion.chunk","choices":[{"index":0,"delta":{"content":"word "},"finish_reason":null}]}
  … (всего 101 content-чанк: round((2200−200)/20)+1) …
data: {"object":"chat.completion.chunk","choices":[{"index":0,"delta":{},"finish_reason":"stop"}]}
data: {"object":"chat.completion.chunk","choices":[],"usage":{"completion_tokens":101}}
data: [DONE]
```

### B. `error` со статусом 429

```http
POST /v1/chat/completions HTTP/1.1
Authorization: Bearer <api_key>
Content-Type: application/json

{ "model": "stub-model", "stream": true,
  "messages": [{ "role": "user", "content": "... [[faker:type=error;status=429]]" }] }
```

```
HTTP/1.1 429 Too Many Requests
Content-Type: application/json

{ "error": { "message": "Rate limit exceeded", "type": "rate_limit_error", "code": "rate_limit_exceeded" } }
```
(ЛК может перемапить статус и переписать тело; клиент сверяет лишь класс «ошибка».)
